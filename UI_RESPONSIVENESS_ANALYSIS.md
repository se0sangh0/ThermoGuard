# UI 반응성 및 성능 개선 분석

## 1. 분석 목적

현재 제품형 GUI에서 버튼을 클릭하거나 화면을 갱신할 때 느려질 수 있는 코드를 확인하고, 문제를 다음 세 가지로 구분해 정리한다.

- UI 자체의 문제
- 백엔드 처리 성능 문제
- 무거운 백엔드 작업을 UI 스레드가 직접 호출해서 발생하는 문제

분석 결과, 가장 큰 원인은 화면 디자인이 아니라 **파일 검색, ExifTool 실행, 온도 분석 및 이미지 생성을 Tkinter UI 스레드에서 동기 실행하는 구조**이다.

---

## 2. 핵심 결론

### 현재 구조

```text
Tkinter UI 스레드
  └─ 주기 갱신 또는 새로고침 버튼
      ├─ 최신 파일 검색 및 정렬
      ├─ ExifTool 실행
      ├─ 온도 행렬 생성 및 로드
      ├─ ROI 통계·핫스팟·이상 상태 분석
      ├─ 열화상 오버레이 생성
      ├─ 가시광 이미지 로드
      └─ 이미지 축소 및 화면 표시
```

이 작업이 끝날 때까지 Tkinter가 입력과 화면 다시 그리기를 처리하지 못하므로 창 이동, 버튼 클릭, 종료 반응이 늦어질 수 있다.

### 권장 구조

```text
카메라 캡처 스레드
  ↓ 촬영 파일 경로 전달
ExifTool·분석 전용 작업 스레드
  ↓ 분석 완료 결과 생성
Result Snapshot / Queue
  ↓ 최신 결과 전달
Tkinter UI 스레드
  └─ 수치, 로그, 이미지 표시만 수행
```

UI 버튼 콜백은 작업을 요청한 뒤 즉시 반환해야 한다. OpenCV·NumPy 처리는 작업 스레드에서 수행하고, Tkinter 위젯 변경과 `ImageTk.PhotoImage` 생성만 UI 스레드에서 수행한다.

## 3. 주요 성능 문제

### 3.1 화면 갱신에서 무거운 작업을 동기 실행

대상 파일:

- `thermal_monitoring/tools/product_dashboard.py`
- `refresh_now()` 약 262행
- `_latest_pair()` 약 278행
- `_process_pair()` 약 292행

`refresh_now()`는 `root.after()`로 호출되므로 Tkinter 메인 스레드에서 실행된다. 하지만 내부에서 다음 작업을 모두 직접 실행한다.

- 데이터셋 파일 검색
- NPY 생성
- ROI 통계 계산
- 상태 판정
- 핫스팟 검출
- 오버레이 이미지 생성
- 가시광 이미지 로드
- PIL 리사이즈 및 화면 표시

발생 가능한 증상:

- 창 이동 및 크기 변경이 끊김
- 버튼 클릭 후 화면이 늦게 반응함
- Windows에서 일시적으로 응답 없음처럼 보임
- 사용자가 새로고침 버튼을 반복해서 누르게 됨

개선 방법:

1. 분석 전용 `ThreadPoolExecutor(max_workers=1)`를 둔다.
2. `refresh_now()`는 작업이 진행 중인지 확인한 후 분석 작업만 제출한다.
3. 작업 스레드는 파일 검색, ExifTool, NumPy/OpenCV 분석을 수행한다.
4. 완료 결과는 `root.after(0, ...)`로 UI에 전달한다.
5. UI 스레드는 결과 표시만 수행한다.

---

### 3.2 ExifTool 실행에 따른 장시간 UI 정지 가능성

대상 파일:

- `thermal_monitoring/capture/thermal_utils.py`
- `extract_from_jpeg()` 약 79행

열화상 JPG 한 장을 처리할 때 ExifTool 프로세스를 두 번 실행한다.

```text
1차 호출: 메타데이터 추출, timeout 30초
2차 호출: RawThermalImage 추출, timeout 30초
```

NPY 파일이 없으면 `product_dashboard.py`의 `_latest_pair()`가 이 함수를 UI 스레드에서 직접 호출한다. ExifTool이 느리거나 응답하지 않으면 화면이 오랫동안 멈출 수 있다.

개선 방법:

- ExifTool 처리를 분석 작업 스레드로 이동한다.
- 가능하면 캡처 직후 NPY 파일을 미리 생성한다.
- UI는 변환이 완료된 결과만 읽는다.
- 장기적으로 ExifTool `-stay_open` 방식으로 프로세스를 재사용한다.
- Windows 콘솔 숨김 처리는 화면 깜빡임 개선이며 처리 속도 개선과는 별개로 본다.

---

### 3.3 데이터셋 전체 검색 및 정렬

화면 갱신 시 다음 형태로 모든 JPG 파일을 검색하고 정렬한다.

```python
thermal_files = sorted(
    p for p in dataset.glob("*.jpg") if "_visual" not in p.name
)
```

같은 검색 방식이 ROI 설정과 캘리브레이션에서도 사용된다.

초기에는 문제가 작지만, 데이터 파일이 수천 개 이상 쌓이면 갱신할 때마다 디렉터리 I/O와 정렬 비용이 증가한다.

개선 방법:

- 캡처 성공 시 최신 파일 경로를 메모리에 저장한다.
- 캡처 스레드가 분석 대기열에 파일 경로를 직접 전달한다.
- 필요하면 `latest_capture.json` 같은 최신 상태 인덱스를 둔다.
- 데이터셋을 날짜별 하위 폴더로 분리한다.
- UI가 전체 데이터 폴더를 반복 검색하지 않도록 한다.

---

### 3.4 이미지 분석과 렌더링 비용

`_process_pair()`와 `_show_image()`에서는 다음 처리가 수행된다.

- NPY 로드 및 `float64` 변환
- ROI percentile 계산
- 연결 요소 기반 핫스팟 검출
- OpenCV 이미지 로드
- Homography 좌표 변환
- 오버레이 생성
- OpenCV BGR → RGB 변환
- PIL 이미지 생성 및 LANCZOS 리사이즈
- Tkinter `PhotoImage` 생성

개선 방법:

작업 스레드에서 수행할 항목:

- NPY 로드와 온도 분석
- ROI·핫스팟 계산
- 상태 판정
- 오버레이 생성
- 표시 크기에 맞춘 이미지 축소

UI 스레드에서 수행할 항목:

- `ImageTk.PhotoImage` 생성
- Label 이미지 변경
- 상태 수치와 로그 갱신

주의: Tkinter 위젯과 `PhotoImage`는 작업 스레드에서 직접 변경하지 않는다.

### 3.5 프로그램 종료 시 UI 정지 가능성

대상 파일:

- `thermal_monitoring/tools/product_dashboard.py`의 `on_close()` 약 402행
- `thermal_monitoring/capture/capture.py`의 `stop()` 약 82행

현재 `running → closing → closed` 상태는 구분되어 있다. 다만 UI 스레드에서 `capture.stop()`을 호출하고, 내부에서 다음과 같이 캡처 스레드 종료를 기다린다.

```python
self._thread.join(timeout=self.interval + 5)
```

캡처 주기가 10초라면 종료 버튼을 누른 후 최대 약 15초 동안 창이 멈춰 보일 가능성이 있다.

권장 종료 흐름:

```text
종료 클릭
  ↓
lifecycle = closing
  ↓
버튼과 갱신 타이머 비활성화
  ↓
캡처 스레드에 중단 요청
  ↓
root.after()로 종료 여부 반복 확인
  ↓
lifecycle = closed
  ↓
창 제거
```

UI 스레드에서는 장시간 `join()`으로 기다리지 않는 것이 좋다.

### 3.6 ROI 및 캘리브레이션 창 초기 로딩

ROI 편집기는 버튼 클릭 직후 큰 가시광 이미지를 읽고 색상 변환과 리사이즈를 수행한다. 캘리브레이션도 최신 파일 검색과 이미지 로딩을 버튼 호출 경로에서 수행한다.

발생 가능한 증상:

- 버튼 클릭 후 편집 창이 늦게 나타남
- 큰 이미지에서 첫 화면이 잠시 멈춤
- 작업 중 버튼을 다시 눌러 중복 창이 열릴 수 있음

개선 방법:

- 편집 창을 먼저 표시한다.
- `이미지를 불러오는 중...` 상태를 표시한다.
- 이미지 로드와 축소는 작업 스레드에서 처리한다.
- 완료된 결과를 UI 스레드에서 캔버스에 표시한다.
- 로딩·실행 중에는 관련 버튼을 비활성화한다.

## 4. UI 자체가 주원인이 아닌 부분

다음 기능은 이미 백그라운드 처리되어 있어 일반적인 화면 멈춤의 직접 원인은 아니다.

- 카메라 통신 및 캡처 스레드
- 열화상·가시광 HTTP 요청 작업 풀
- 비동기 연결 확인 `_check_connection_async()`

다만 다음 작업이 동시에 실행되면 CPU, 디스크 또는 네트워크 자원 경합으로 전체 처리 속도가 느려질 수 있다.

- GUI와 별도 파이프라인 프로그램 동시 실행
- 동일한 이미지에 대한 ExifTool 중복 실행
- 데이터 정리 작업과 UI 분석의 동시 파일 접근
- 지나치게 빈번한 파일 로그 기록

이는 UI 디자인 문제가 아니라 백엔드 자원 사용과 실행 구조 문제이다.

## 5. 10~30초 화면 갱신 정책과 분석 누락 문제

현재 제품 UI는 최신 파일 한 장을 찾아 분석하는 방식이다. 캡처가 10초 간격이고 화면 갱신이 20초 간격이라면 중간 프레임이 분석되지 않을 수 있다.

가능한 문제:

- 중간 프레임의 이상 상태 누락
- 이상 감지 횟수 부정확
- 시퀀스 정상 완료 횟수가 실제 캡처 횟수와 달라짐

권장 정책:

```text
백엔드 분석
  └─ 캡처된 모든 프레임을 순서대로 처리
      ├─ 이상 여부 저장
      ├─ 통계 갱신
      └─ 로그 기록

UI 화면
  └─ 10~30초마다 가장 최근 분석 결과만 표시
```

화면을 매 프레임 갱신하지 않더라도 백엔드 분석은 모든 프레임에 대해 수행해야 한다.

## 6. 권장 작업 상태 관리

중복 작업과 오래된 결과 표시를 막기 위해 다음 상태를 관리하는 것이 좋다.

```text
refresh_requested     새 화면 결과 요청 여부
analysis_in_progress  현재 분석 작업 실행 여부
pending_latest        작업 중 새 캡처가 들어왔는지 여부
generation_id         오래된 분석 결과 폐기용 번호
```

동작 예시:

```text
새로고침 요청
  ├─ 분석 중 아님 → 작업 시작
  └─ 분석 중임     → pending_latest만 설정

분석 완료
  ├─ 최신 generation → 화면 반영
  └─ 오래된 generation → 결과 폐기

pending_latest 있음
  └─ 가장 최신 파일 한 건으로 다음 작업 시작
```

## 7. 성능 측정 로그 권장 항목

개선 전후를 객관적으로 비교하려면 단계별 시간을 기록해야 한다.

| 측정 항목 | 의미 |
|---|---|
| 카메라 요청 시간 | 열화상·가시광 이미지 수신 속도 |
| ExifTool 메타데이터 시간 | 외부 프로세스 첫 번째 호출 비용 |
| RawThermalImage 추출 시간 | 외부 프로세스 두 번째 호출 비용 |
| NPY 로드 시간 | 배열 파일 읽기 및 변환 비용 |
| ROI 분석 시간 | 통계·percentile 계산 비용 |
| 핫스팟 검출 시간 | OpenCV 연결 요소 분석 비용 |
| 오버레이 생성 시간 | 이미지 로드·좌표 변환·그리기 비용 |
| UI 렌더링 시간 | 이미지 축소·PhotoImage·위젯 갱신 비용 |
| 전체 시퀀스 시간 | 한 프레임의 전체 처리 시간 |
| UI heartbeat 지연 | 메인 이벤트 루프가 멈춘 시간 |

운영 로그에는 평균만 남기지 말고 `p50`, `p95`, 최댓값을 함께 집계하는 것이 좋다.

## 8. 개선 우선순위

### P0 — 우선 적용 필요

1. `refresh_now()`의 파일 검색, ExifTool, 분석을 작업 스레드로 이동
2. 종료 시 UI 스레드에서 `join()`하지 않도록 변경
3. 새로고침 버튼 중복 요청 방지
4. 백엔드는 모든 캡처를 분석하고 UI는 최신 결과만 표시하도록 분리

### P1 — 데이터 증가 전에 적용

1. 전체 데이터셋 검색과 정렬 제거
2. 캡처에서 분석기로 파일 경로 직접 전달
3. ROI와 캘리브레이션 이미지 비동기 로딩
4. 로딩 상태 표시와 작업 중 버튼 비활성화
5. 파일 생성·분석·정리 작업 간 동시 접근 보호

### P2 — 추가 최적화

1. Homography 행렬을 파일 변경 시에만 다시 로드하도록 캐시
2. 큰 이미지를 가능한 이른 단계에서 화면 크기로 축소
3. ExifTool 프로세스 재사용
4. 로그가 많아질 경우 `QueueHandler` 기반 비동기 파일 기록

## 9. 최종 보고 요약문

> UI 구성 자체보다 UI 스레드에서 이미지 검색, ExifTool 변환, 온도 분석 및 오버레이 생성을 동기 실행하는 구조가 반응성 저하의 주요 원인으로 확인되었다. 카메라 캡처와 연결 확인은 이미 백그라운드 처리되어 있으나, 주기 갱신과 프로그램 종료 과정에는 UI 블로킹 가능성이 있다. 분석 전용 작업 스레드와 결과 큐를 도입하고, UI는 10~30초마다 완성된 최신 결과만 표시하도록 분리하는 개선이 필요하다. 또한 데이터 증가에 대비해 매 갱신 시 전체 폴더를 검색·정렬하는 방식을 최신 파일 인덱스 또는 작업 큐 방식으로 변경해야 한다.

## 10. 최종 판단

- 화면 레이아웃을 전면 수정해야 하는 문제는 아니다.
- UI와 분석 엔진 사이의 실행 구조 개선이 핵심이다.
- 가장 먼저 주기 갱신과 프로그램 종료 과정의 UI 블로킹을 제거해야 한다.
- 화면은 10~30초마다 갱신하되, 이상 감지 백엔드는 모든 캡처 프레임을 처리해야 한다.
